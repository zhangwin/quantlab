# 三项目融合概要设计

Qlib（量化基础设施）+ Kronos（K线基础模型）+ RD-Agent（自动化研发Agent）

---

## 一、系统定位

| 项目 | 在系统中的角色 |
|------|---------------|
| Qlib | 数据底座 + 因子引擎 + 回测基准。提供统一的行情数据、Alpha158因子集、交易日历 |
| Kronos | 预测核心。Transformer基础模型，输入历史K线，输出未来K线预测 |
| RD-Agent | 离线研发引擎。LLM驱动的进化循环，自动发现与趋势信号互补的因子 |

三者关系：Qlib 提供数据和评估框架，Kronos 和 RD-Agent 分别从不同维度产出信号，最终融合交易。

---

## 二、核心交易逻辑

**T+1 开盘价交易制度：**

1. 每个交易日 T 收盘后，获取截至 T 日的全部历史数据
2. 三条管线各自产出选股信号
3. 信号融合 → 生成买卖订单（含目标价）
4. T+1 日开盘：以开盘价判断是否成交
   - 买入：开盘价 ≤ 目标价 → 成交，否则放弃
   - 卖出：非跌停即成交
5. 风控模块全程监控

## 四、三条管线设计

### 管线A：Alpha158 + LightGBM

| 项目 | 说明 |
|------|------|
| 数据 | Qlib Alpha158 因子集（158个量价统计特征） |
| 模型 | LightGBM（GBDT） |
| 训练 | 滚动训练，每20个交易日用最近3年数据重训 |
| 预测目标 | T+1 日收益率 |
| 信号频率 | 日频 |
| 信号特性 | 动量/反转/波动率等统计规律，覆盖面广 |

### 管线B：Kronos 滚动微调

| 项目 | 说明 |
|------|------|
| 数据 | 最近30天 OHLCV K线 |
| 模型 | Kronos 预训练模型（冻结）+ 最后2层微调 |
| 训练 | 每日轻量微调：30天数据、2-3 epoch、约5分钟 |
| 预测目标 | 未来5天 K 线（OHLCV） |
| 信号输出 | return_1d（次日收益，用于日频融合）、return_5d（5日收益）、uncertainty（预测分歧度） |
| 信号特性 | K线形态延续，捕捉序列模式 |
| 推理方式 | predict_batch 全市场一次性推理（~15分钟） |

**微调策略：**
- 每日从预训练权重重新出发（避免误差累积）
- 冻结预训练参数，只解冻最后2层 Transformer + 输出头
- 10次采样取均值作为预测值，标准差作为不确定性度量

### 管线C：RD-Agent 进化因子

| 项目 | 说明 |
|------|------|
| 运行频率 | 离线，每周末运行一次进化循环 |
| 进化目标 | 定向生成与管线A/B互补的因子 |
| 因子方向 | 均值回复、波动率异常、流动性变化（非趋势类） |
| 过拟合控制 | 反馈中加入 train IC - test IC 差值惩罚 |
| 集成方式 | 进化产出因子代码 → 嵌入日频管线在线计算 |

**三管线互补关系：**

```
       趋势/动量维度          形态/序列维度         均值回复/波动率维度
            │                     │                      │
         管线A                  管线B                   管线C
     Alpha158+LightGBM       Kronos                  RD-Agent
            │                     │                      │
            └──────── 正交融合 ────┘──────────────────────┘
```

## 五、信号融合流程

```
管线A 信号（日频 T+1 收益预测）
管线B 信号（Kronos return_1d + uncertainty）
管线C 信号（均值回复因子截面值）
      │
      ▼
Step 1: Horizon 对齐
      · 管线A：直接使用（已是日频）
      · 管线B：使用 return_1d（Kronos 对 T+1 的预测分量）
      · 管线C：直接使用（已是日频截面值）
      │
      ▼
Step 2: 截面 Rank 归一化
      · 每个信号在当日截面内排序，映射到 [0, 1]
      · 消除不同信号的量纲和分布差异
      │
      ▼
Step 3: Expanding IC 动态加权
      · 用历史已实现的（信号, 次日收益）计算各信号的滚动 Rank IC
      · IC 高的信号权重大，IC ≤ 0 的信号权重归零
      · 窗口：最近60个交易日的 IC 均值
      · 严格 expanding window：只用到当天为止的历史，无前视偏差
      │
      ▼
Step 4: 不确定性惩罚
      · Kronos 10次采样标准差大的股票，融合得分下调
      · 目的：对"模型自己也没把握"的标的降低仓位
      │
      ▼
最终融合信号（pd.Series: symbol → score）
```

## 七、数据时间隔离方案

**原则：任何模块在 T 日只能看到 ≤ T 日的数据。T+1 开盘价仅在成交判定时使用。**

### 每日数据更新

```
回测模式：回测启动前一次性将数据更新到 backtest_end
实盘模式：每个交易日 15:30 后通过定时任务增量更新

更新流程：
  1. 读取 calendars/day.txt 获取最新日期 last_date
  2. 若 last_date >= 目标日期 → 跳过
  3. 调用 Qlib Yahoo Collector 增量下载 (last_date, end_date] 的行情
  4. DumpDataUpdate 追加写入 .bin 文件（不重写历史）
  5. 更新交易日历和成分股列表
  6. 重新初始化 Qlib
```

```
时间线：
... T-31  T-30  ...  T-1    T      T+1    T+2  ...  T+5
     │←── Kronos 微调窗口 ──→│
     │←────── Kronos 推理输入 ──→│→ 预测 →│          │
                                  │       │          │
                             信号生成   成交判定   预测终点
                             (收盘后)  (开盘时)
```

| 环节 | 可用数据范围 | 不可用 |
|------|-------------|--------|
| Kronos 微调 | T-30 ~ T 的 OHLCV | T+1 及之后 |
| Kronos 推理 | T-lookback ~ T（输入），T+1 ~ T+5（输出目标） | 推理时无实际 T+1 数据 |
| Alpha158 特征计算 | ≤ T 日的全部行情 | T+1 及之后 |
| Alpha158 训练标签 | ≤ T-2 日（标签需 T+2 的 close） | T-1, T 日的标签无法计算 |
| 信号融合 IC 权重 | 历史已实现的（信号, 收益）对 | 当日及之后的收益 |
| 成交判定 | T+1 开盘价 | 仅用于执行，不回流到信号 |

## 八、风控体系

**三级递进式风控：**

```
Level 1  单票级
    触发：持仓从最高价回撤 ≥ 8%
    动作：下一交易日强制卖出
    目的：控制个股黑天鹅

Level 2  行业级
    触发：单行业市值占比 > 总持仓 30%
    动作：卖出该行业中仓位最小的票
    目的：防止行业集中暴露

Level 3  组合级
    触发：组合净值从高水位回撤 ≥ 10%
    动作：全部清仓，暂停交易 5 个交易日
    目的：系统性风险保护
```

## 九、评估维度

| 维度 | 指标 | 目的 |
|------|------|------|
| 绩效 | 年化收益、夏普比率、最大回撤、Calmar | 整体盈利能力 |
| 交易质量 | 买入成交率、总成本拖累、日均换手率 | 执行层效率 |
| 信号质量 | 各管线 IC 均值、ICIR、IC 衰减曲线（T+1~T+10） | 判断各信号最佳持仓周期 |
| 环境适应性 | 牛/熊/震荡市分别的收益和回撤 | 诊断系统弱点 |
| 融合效果 | 融合信号 vs 各单管线信号的绩效对比 | 验证融合是否有增益 |
