二、主交易链路

1

数据校验

用 random_check_qlib_data.py 确认 A_data 完整、calendar 和 bin header 对齐、近期覆盖率正常。

2

Kronos 推理

用 exp_pipeline_daily-new.py 生成每日 embedding、pred_ohlc、selector features、fwd labels。

3

融合排序

先以 3-feature Kronos 为 baseline，再加入因子、板块资金流和相对偏离 selector。

4

Agentic 审查

输出结构化 ReviewDecision，先只记录，回测通过后才允许影响仓位。

5

执行与复盘

用真实费用、T+1、停牌涨跌停规则生成账本，再做分年/分行业/分信号归因。

当前最重要的工程边界

不要把 agent、融合选择器、资金流处理写死在 EXP 脚本里。
EXP 脚本只负责组织输入、调用模块、保存报告。核心逻辑必须沉到 quantlab/data、quantlab/signal、quantlab/agentic、quantlab/execution 这些可复用包里。

当前需要严格避免

  • 让 LLM 直接替代 Kronos/factor selector。
  • 把资金流数据写入原始 A_data/features。
  • 继续复用旧坏数据下的 pipeline 产物。
  • 只看 IC，不看真实费用后的 ledger。
  • 未冻结 prompt/model/data snapshot 就回测 LLM 决策。

四、Agentic 模块设计

输入：ReviewContext

由 selector 输出和辅助数据构建，保持可序列化。

  • date
  • top-N candidates
  • industry map
  • sector flow features
  • portfolio context
  • market state

审查器：Reviewers

当前先用规则型，后续加入 LLM 型。

  • LiquidityReviewer
  • SectorFlowReviewer
  • ConcentrationReviewer
  • FundamentalReviewer
  • BullBearDebateReviewer

输出：ReviewDecision

实验和执行只读取结构化字段。

  • action: approve / reduce / veto / review
  • position_scale
  • veto_symbols
  • final_candidates
  • findings

五、实验路线

P0 基线复核

重跑 2022-2026 Kronos-base 后，先重建真实 ledger。输出新的 Sharpe、maxDD、分年收益和交易清单。

P1 参数稳健性

只扫 K 和 hold 的小网格，确认当前生产参数是否处于稳健平台，不做过拟合式调参。

P2 相对偏离

把绝对 selector 改成相对当日市场中位数的偏离，降低市场整体情绪噪音。

P3 板块资金流

先监控和解释，再验证弱板块剔除、强板块加权是否改善 maxDD 或 hit rate。

P4 Agentic Overlay

先记录审查结果，不改交易；通过回测后再允许 veto/reduce/position_scale 进入执行链路。

P5 ML Selector

用 embedding、Kronos 特征、因子、资金流和路径衍生特征做 walk-forward selector。
