QuantLab Feature Article

从选股信号到可解释交易决策：Agentic 复核层与板块资金流新特性

QuantLab 的新增模块把 Kronos、因子分析、板块资金流和规则化 Agent 复核连接起来。 它不是把大模型直接放进回测核心，而是在量化排序之后增加一个可记录、可审计、可逐步升级的决策层， 用来解释候选股票、发现风险、控制组合暴露，并为后续基本面分析和多空辩论预留接口。

核心模块quantlab.agentic
数据增强Sector Fund Flow
接入位置选股后、下单前
当前形态规则化、可回测
扩展方向基本面、辩论、复盘

为什么需要这一层

纯量化信号解决的是“哪些股票更可能有超额收益”，但真实交易还需要回答“这个组合是否拥挤、是否缺流动性、是否处在资金流明显恶化的板块里、是否应该降低仓位”。 新增的 Agentic 复核层把这些问题从实验脚本中抽离出来，形成一个独立、稳定、可复用的模块。

Alpha

信号负责排序

Kronos 预测、Alpha 因子、后续机器学习排序模型负责生成候选池和分数。它们输出的是候选股票、rank、score、预测特征和选中标记。

Agentic

复核负责解释和过滤

Agentic 复核层读取候选池和组合上下文，输出结构化发现、否决股票、降权股票、整体仓位缩放和最终候选名单。

Data

资金流负责补充横截面语义

板块资金流把“股票处在哪个行业/概念、该板块近期资金是否持续流入”引入决策，使选股不只看个股历史形态。

总体架构

新特性位于“候选股票排序”和“交易执行”之间。这样做有两个好处：第一，回测主链路仍然清晰； 第二，未来即使加入 LLM 基本面分析、多空辩论和复盘，也不会污染底层数据、信号和执行模块。

Data Layer
Qlib A 股日频数据

OHLCV、amount、factor、calendar、instruments，所有查询受交易日和 anchor date 约束。

Sector 板块资金流

行业/概念资金流下载、清洗、标准化，并对齐 Qlib 日历和行业映射。

Signal Layer
Kronos K 线预测

基于 lookback 窗口预测未来 OHLCV 分布，形成趋势、反转、波动和不确定性特征。

Factor 因子分析

Alpha158、技术面因子、市场中性特征、行业相对强弱等用于横截面排序。

Fusion Selector
Rank 候选池生成

Kronos + 因子分数融合，输出 top-N 候选和 top-K 初选组合。

Meta 上下文打包

把 symbol、rank、score、行业、均额、last close、特征列转换为 ReviewContext。

Agentic Review
Rules 确定性 reviewer

流动性、板块资金流、行业集中度等规则先落地，保证可回测和可解释。

Future LLM reviewer

后续可增加基本面、新闻、多空辩论、复盘 reviewer，但必须遵循同一结构化接口。

Decision
Action 结构化决策

approve、reduce、veto、review，并给出 position_scale、veto_symbols、final_candidates。

Record 复核记录

输出 JSON/CSV，便于每日归档、回测复盘、策略对比和人工审阅。

Execution & Eval
Trade T+1 执行

最终候选和仓位缩放进入交易执行层，保持下单逻辑和风控逻辑独立。

Backtest 实验评估

对比无复核、仅流动性复核、资金流复核、组合集中度复核等实验版本。

日频决策流程

当前实现适合放在每日推理完成之后运行。它不替代 Kronos 或因子模型，而是读取模型输出的候选池，对候选组合做交易前复核。

1

数据更新

更新 Qlib 日频数据、日历、股票池和行业映射；外部板块资金流写入扩展目录。

2

信号推理

Kronos 和因子管线生成预测特征、横截面分数和候选股票排序。

3

候选池打包

把 meta 文件、行业映射、均额、板块资金流合并为 ReviewContext。

4

Reviewer 复核

Liquidity、SectorFlow、Concentration 等 reviewer 生成结构化 findings。

5

决策聚合

AgenticReviewEngine 聚合发现，生成 action、仓位缩放、否决名单和最终候选。

6

交易执行

执行层只消费最终候选和仓位约束，不关心每个 reviewer 的内部细节。

7

复盘归档

保存 JSON/CSV，用于解释每日交易、复盘误判和训练后续人工/LLM reviewer。

Agentic 模块的接口边界

这次设计的关键点是“独立模块，而不是写在 EXP 脚本里”。实验脚本可以调用它，但模块本身不依赖某一个实验。

文件 职责 为什么重要
quantlab/agentic/schemas.py 定义 CandidateStock、ReviewContext、ReviewFinding、ReviewDecision 等结构。 让选股管线、复核器、执行层通过稳定数据结构解耦。
quantlab/agentic/reviewers.py 实现 BaseReviewer、LiquidityReviewer、SectorFlowReviewer、ConcentrationReviewer。 当前 reviewer 是规则化的，因此可以进入历史回测，不依赖实时人工判断。
quantlab/agentic/engine.py 运行所有 reviewer，聚合 findings，生成最终决策。 把多条复核意见压缩为交易层能消费的结构化结果。
quantlab/agentic/io.py 从 pipeline meta、行业映射、板块资金流构造 ReviewContext。 解决实验输出和独立 Agentic 模块之间的数据适配问题。
quantlab/agentic/review_cli.py 提供日频复核命令行入口，输出 JSON 和 CSV。 便于先跑离线分析，再决定是否接入正式回测。

四类复核决策

AgenticReviewEngine 的输出不是自然语言建议，而是结构化动作。自然语言解释会保留在 finding message 里，但交易层只读取稳定字段。

approve

候选组合没有触发重要风险，保持原始 top-K 和默认仓位。

reduce

组合存在板块资金流走弱、行业集中等风险，保留交易但降低整体或局部仓位。

veto

个股触发严重流动性或其他硬约束，从最终候选中移除，并可用后备候选替换。

review

发现中等风险但不足以自动否决，输出审阅名单，用于人工或更高阶 Agent 复核。

板块资金流如何与现有数据对齐

数据不写入 Qlib 主目录

板块资金流来自外部数据源，字段语义和更新机制都不同于 OHLCV。 因此建议放在 quantlab/A_data_ext/sector_flow，避免污染 quantlab/A_data 的标准 Qlib bin 数据。

  • normalized/sector_flow_daily：原始资金流标准化结果。
  • features/sector_flow_features：滚动排名、z-score、持续性、冷热标签。
  • industry_map.csv：股票到行业的映射桥梁。

对齐原则

所有资金流特征必须按 Qlib 交易日历对齐，并且在 T 日复核时只能使用 T 日及以前的数据。 这与主回测的 anchor date 原则一致，避免资金流特征产生前视偏差。

  • 日期对齐：使用 calendars/day.txt 作为唯一交易日基准。
  • 行业对齐：通过 industry_map.csv 将股票映射到板块。
  • 特征对齐：T 日候选股票只读取 T 日可见资金流特征。

一次日频复核的输入与输出

命令行示例

python quantlab/agentic/review_cli.py \
  --date 2026-05-07 \
  --meta-path quantlab/experiments/reports/pipeline_daily_base/2026-05-07_meta.csv \
  --industry-map quantlab/industry_map.csv \
  --sector-flow quantlab/A_data_ext/sector_flow/features/sector_flow_features.csv \
  --output-dir quantlab/experiments/reports/agentic_review \
  --target-k 5 \
  --candidate-n 20

输出文件

复核结果会保存为结构化记录，适合后续做批量统计和回测对比。

  • *_decision.json：完整 ReviewDecision，包括 action、position_scale、final_candidates。
  • *_findings.csv：每个 reviewer 的发现，便于人工查看和聚合分析。
  • *_final_candidates.csv：交易层可消费的最终候选名单。

它和 LLM Agent 的关系

当前版本故意先规则化。 规则化 reviewer 可以直接进入历史回测，能回答“加入这一层后收益、回撤、换手、胜率是否改善”。 后续 LLM reviewer 可以作为同一接口下的插件加入，但它应该先用于解释、审阅、复盘和人工辅助，而不是直接替代可回测信号。
Bull / Bear

多空辩论

对同一个候选股票生成看多与看空论点，最终落到结构化 finding，而不是停留在自然语言评论。

Fundamental

基本面复核

接入财报、估值、盈利质量、公告和新闻摘要，作为交易前的审阅层，而不是数据泄漏的 alpha 特征。

Memory

复盘记忆

记录每次 veto/reduce 的结果，分析哪些 reviewer 真正降低了回撤，哪些只是噪声。

实验路线

阶段 目标 判断标准
P0 基线 先跑 2022 至今的 Kronos + 因子融合基线，不开启 Agentic 复核。 确认数据覆盖、候选池稳定性、收益/回撤/换手的可复现基准。
P1 规则复核 开启流动性、行业集中度和板块资金流 reviewer。 观察是否降低极端亏损、无效交易和冷板块暴露。
P2 参数稳健性 扫描 target_k、candidate_n、position_scale、资金流阈值。 不能只看单次收益，要看多个窗口下的收益回撤比和换手变化。
P3 解释质量 统计 reviewer finding 与后续收益、回撤、成交失败之间的关系。 保留有预测价值的 reviewer，移除只增加噪声的规则。
P4 LLM 扩展 在同一接口下加入基本面、新闻、多空辩论和复盘 reviewer。 LLM 输出必须结构化、可归档、可抽样审计，不能破坏回测边界。

设计原则总结

信号与决策分离

Kronos 和因子模型负责发现机会，Agentic 层负责复核和约束交易。

外部数据隔离

板块资金流放在扩展目录，通过日期和行业映射对齐，不污染 Qlib 主数据。

先可回测再智能化

当前 reviewer 先规则化，未来 LLM reviewer 也要遵守结构化接口和时间边界。

输出可审计

每次 approve、reduce、veto、review 都必须留下 finding 和最终候选记录。
